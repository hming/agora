1. AGORA 解决的根本问题是什么？

3. 怎么把自己的 Agent 接入进来？

## 一、理解问题：为什么多 Agent 协调很难？

### 信息 vs. 公共知识

假设你有两个 Agent，A 和 B，它们都在处理同一份文档。你告诉 A："B 已经完成了摘要任务。"你也告诉 B 同样的话。

现在 A 和 B 都**知道**对方完成了。但这还不够。

A 不知道 B 是否知道 A 已经知道这件事。B 不知道 A 是否收到了通知。在这种不确定性下，双方都会采取保守策略——等待更多确认，或者重复对方的工作。

这是 Michael Chwe 在《理性的仪式》中揭示的核心问题：

> **协调失败的根源不是"缺少信息"，而是"缺少公共知识"。**

**互知（Mutual Knowledge）**：A 知道 X，B 知道 X。
**公共知识（Common Knowledge）**：A 知道 X，B 知道 X，A 知道 B 知道 X，B 知道 A 知道 X，且这个链条无穷延伸。

只有公共知识才能支撑可靠的协调。人类社会用**仪式**来生产公共知识：广场上的公开集会，所有人都看到所有人都在场。AGORA 把这个机制搬进了多 Agent 系统。

### 现有框架的问题

LangGraph、AutoGen、CrewAI 的标准做法是引入一个中心调度节点（Orchestrator / Supervisor）：

```
[Orchestrator]
    ├── Agent A
    ├── Agent B
    └── Agent C
```

这个架构有三个根本性问题：

**1. 单点故障**：Orchestrator 挂了，整个系统停。
**2. 隐式协调**：Agent 之间的协调状态存在 Orchestrator 内存中，其他 Agent 看不到，无法验证。
**3. 框架耦合**：你的 Agent 必须用框架提供的 API 定义，换个模型或语言就要重写。

AGORA 的出发点是：**协调状态应该是公开的、可验证的、任何 Agent 都能独立读取的。**

## 二、AGORA 的解法

### 广场（Agora）：公共知识的物理载体

AGORA 的核心是一个 **Append-Only 的公共日志**，基于 Redis Stream 实现。所有协调事件都写入这里：

```
时间戳  Agent     消息类型          内容
──────────────────────────────────────────────
t=1    system    GOAL_RECEIVED     "分析竞品并输出报告"
t=2    system    GOAL_DECOMPOSED   [task_1, task_2, task_3]
t=3    agent-a   AGENT_JOINED      capabilities: ["search"]
t=4    agent-b   AGENT_JOINED      capabilities: ["write"]
t=5    agent-a   TASK_CLAIMED      task_1: "搜集竞品数据"
t=6    agent-b   TASK_CLAIMED      task_2: "分析市场定位"
t=7    agent-a   TASK_DONE         task_1: "已收集 5 家竞品数据"
t=8    system    TASK_UNBLOCKED    task_3: "撰写最终报告"（依赖 task_1 已完成）
...
```

任何 Agent 在任何时刻加入，都能读到完整的协调历史。没有任何信息在私有信道传递。

### Epoch：仪式期，公共知识的生产时刻

每隔 30 秒，系统触发一次"仪式"：

```
1. EPOCH_START   → 所有 Agent 收到"新纪元开始"的信号
2. STATE_PUBLISH → 每个 Agent 公开广播自己当前的状态
3. ACK           → 每个 Agent 逐一确认"我看到了其他每个 Agent 的状态"
4. LEADER_ELECTED → 确定性 Leader 选举（ID 最小者当选）
```

关键在第 3 步：当 Agent A 发布 ACK("我看到了 B 的状态")，B 也看到了这条 ACK，这就完成了公共知识的闭合——**A 知道 B 知道 A 知道**，链条就此建立。

这是仪式的本质：不是传递信息，而是让所有参与者**共同见证**信息的传递。

### 任务层：声明式 DAG，原子认领

任务以 DAG（有向无环图）形式组织：

```json
[
  {"id": "t1", "description": "搜集数据",   "depends_on": []},
  {"id": "t2", "description": "分析数据",   "depends_on": ["t1"]},
  {"id": "t3", "description": "生成报告",   "depends_on": ["t1", "t2"]}
]
```

`t2` 会在 `t1` 完成后自动解锁，`t3` 在 `t1` 和 `t2` 都完成后自动解锁。

认领使用 **Lua 脚本原子操作**：

```lua
-- 扫描 ready 队列，找第一个 required_capabilities 全部满足的任务
-- 原子地移除并返回——避免两个 Agent 同时认领同一任务
```

每个 Agent 声明自己的能力（`capabilities`），只会看到自己能做的任务。

### 仲裁层：三级升级路径

当 Agent 之间出现分歧，仲裁层按成本递增的顺序介入：

| Level | 机制 | 成本 | 触发场景 |
|-------|------|------|---------|
| 0 | Sticky Convention | 零 | 无分歧，继续沿用惯例 |
| 1 | Leader Election | 零通信 | Epoch 边界自动触发 |
| 2 | Threshold Consensus | 一轮投票 | 需要多数同意的决策 |
| 3 | Full Consensus (Raft) | 多轮 | 极罕见，未实现 |

每一层的决策结果都写入 Agora，成为公共知识。

### 3.5 理解消息流

浏览器界面的消息日志就是 Agora 日志的实时镜像。每条消息的含义：

| 消息类型 | 含义 | 谁发 |
|---------|------|------|
| `GOAL_RECEIVED` | 收到目标 | system |
| `GOAL_DECOMPOSED` | 目标已拆解为子任务 | system |
| `AGENT_JOINED` | Agent 加入，声明能力 | Agent 自身 |
| `TASK_CLAIMED` | Agent 认领了某任务 | Agent 自身 |
| `TASK_DONE` | 任务完成，附带结果 | Agent 自身 |
| `TASK_FAILED` | 任务失败，附带错误（含重试次数）| Agent 自身 |
| `TASK_UNBLOCKED` | 某任务的依赖全部完成，进入 ready | system |
| `EPOCH_START` | 新纪元开始 | system |
| `STATE_PUBLISH` | Agent 广播自己的状态 | Agent 自身 |
| `ACK` | Agent 确认收到了某 Agent 的状态 | Agent 自身 |
| `LEADER_ELECTED` | Leader 选举结果 | system |
| `VOTE_REQUEST` | 发起提案投票 | proposer |
| `VOTE` | 某 Agent 的投票 | Agent 自身 |
| `CONSENSUS_REACHED` | 提案通过 | system |
| `CONSENSUS_FAILED` | 提案未通过 | system |
| `GOAL_COMPLETED` | 目标的所有任务均完成 | system |
| `GOAL_FAILED` | 目标的某任务重试耗尽 | system |
| `AGENT_LEFT` | Agent 离开（正常或超时）| system / Agent |

可以通过 `GET /agora` 获取完整历史，也可以通过 WebSocket 实时订阅：

```javascript
const ws = new WebSocket('ws://localhost:8000/ws')
ws.onmessage = (e) => {
  const msg = JSON.parse(e.data)
  console.log(msg.type, msg.agent_id, msg.payload)
}
```

## 四、扩展 AGORA

### 4.1 添加新的 LLM 提供商

在 `backend/llm/` 下新建文件，继承 `LLMProvider`：

```python
# backend/llm/openai.py
from llm.base import LLMProvider, LLMMessage

class OpenAIProvider(LLMProvider):
    def __init__(self):
        from openai import AsyncOpenAI
        self.client = AsyncOpenAI()  # 从环境变量读取 OPENAI_API_KEY

    async def complete(self, messages: list[LLMMessage], system: str = "") -> str:
        msgs = []
        if system:
            msgs.append({"role": "system", "content": system})
        msgs += [{"role": m.role, "content": m.content} for m in messages]

        resp = await self.client.chat.completions.create(
            model="gpt-4o",
            messages=msgs,
        )
        return resp.choices[0].message.content
```

然后在 `backend/main.py` 的 `_build_llm()` 里接入：

```python
def _build_llm():
    provider = os.getenv("LLM_PROVIDER", "claude").lower()
    if provider == "mock":
        from llm.mock import MockProvider
        return MockProvider()
    if provider == "openai":
        from llm.openai import OpenAIProvider
        return OpenAIProvider()
    from llm.claude import ClaudeProvider
    return ClaudeProvider()
```

设置环境变量后重启即可：

```bash
LLM_PROVIDER=openai make dev
```

## 五、常见问题

### Q：Agent 失败后任务会怎样？

**失败 → 自动重试**（最多 3 次，可通过任务 JSON 的 `max_retries` 字段覆盖）。
**重试耗尽 → 级联清理**：所有依赖此任务的下游任务从等待队列中移除，目标发布 `GOAL_FAILED`。
**Agent 心跳超时 → 同上**：持有的任务也进入重试流程。

### Q：两个 Agent 会同时认领同一任务吗？

不会。认领操作通过 Redis Lua 脚本实现原子性：找到匹配任务、从队列移除、返回——这三步不可分割。

### Q：重启后端会丢失什么？

- **不丢失**：Agora 日志（Redis Stream 持久化）、任务队列
- **会丢失**：Agent Registry（内存状态）、仲裁提案（内存状态）

内部 Agent 会在重启后重新注册。外部 Agent 的心跳会超时，触发自动清理。

### Q：如何调试任务没有被认领的问题？

1. 检查队列状态：`GET /tasks/pending`（返回 ready/waiting/pending 数量）
2. 检查 Agent 能力：`GET /agents`（查看每个 Agent 的 capabilities）
3. 确认任务的 `required_capabilities` 是否有 Agent 能满足
4. 查看 `TASK_UNBLOCKED` 消息是否出现——如果依赖没有完成，任务会卡在 waiting

### Q：如何在生产环境使用？

当前实现是单进程的。扩展建议：
- Redis 使用持久化配置（AOF 或 RDB）
- 后端使用 `gunicorn` 多进程（注意：内存状态如 Registry 需要迁移到 Redis）
- M10 计划中：用 Kafka 替换 Redis Stream，天然支持多 broker 水平扩展

## 六、心智模型总结

把 AGORA 想成**一个广场，而不是一个指挥官**。

- **广场**：所有事件公开发生，任何人都能来，任何人都能看到其他人看到了什么
- **Epoch**：定期举行的全体集会，让"我们都在这里"成为公共知识
- **Agent**：广场上的参与者，自带能力，自主认领任务，自主决定行动
- **任务队列**：广场上公开张贴的任务布告板，先到先得，能者认领

正确的使用姿势：**你负责让 Agent 变聪明，AGORA 负责让 Agent 之间的协调变可靠。**
