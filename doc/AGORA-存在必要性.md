# AGORA 存在必要性

## "一个 Agent 对应一个 Task" 的隐藏前提

这个模型能工作，前提是有人负责以下四件事：

1. **把 task 分配给哪个 agent**（避免两个 agent 同时认领同一个 task）
2. **task B 依赖 task A，A 完成后谁来通知 B 可以开始**
3. **agent 中途挂掉，task 怎么重新分配**
4. **所有 task 都完成了，谁来判断整体目标达成**

这个"有人"，要么是你自己写的中心调度器，要么是 AGORA。

---

## 如果你选择自己写中心调度器

```
你的调度器
  ├── 维护 task 状态表（pending/running/done/failed）
  ├── 分配 task 给 agent（需要并发锁，避免重复分配）
  ├── 监听每个 agent 的心跳（检测失效）
  ├── 失效后重新分配 task（需要知道哪些 task 在该 agent 上）
  ├── 跟踪 DAG 依赖，解锁下游 task
  └── 判断整体目标完成
```

你刚刚重新发明了一个 AGORA，但更脆弱：
- 调度器本身挂掉 → 整个系统停摆
- 调度器重启 → task 状态丢失
- 只有调度器知道全局状态 → 其他 agent 对整体进展一无所知

---

## AGORA 的本质回答

AGORA 不是"让 agent 执行 task"的工具，它是把**协调逻辑从中心节点里抽出来，放进共享日志**。

```
没有 AGORA：  Agent A → [中心调度器] → Agent B
                              ↑
                         单点故障
                         状态不透明
                         重启即丢失

有 AGORA：    Agent A → [广场日志] ← Agent B
                              ↑
                         任何人可读
                         持久化
                         Agent 挂掉不影响日志
```

两者解决的协调问题是一样的，区别在于：**协调状态住在哪里**。

---

## 什么时候真的不需要 AGORA

如果你的场景满足以下全部条件：

- task 之间**没有依赖关系**（全部并行独立）
- **不在乎重复执行**（幂等任务，多做一次没问题）
- **不在乎失败**（agent 挂掉就算了）
- **单次运行**，不需要持久化状态
- **只有一个调用方**，不存在多 agent 竞争

那确实不需要 AGORA，一个简单的 for 循环 spawn 任务就够了。

---

## 结论

AGORA 的价值在于：**任务之间有依赖、有竞争、有失败的时候，你不得不解决协调问题——AGORA 让你不用自己造一个更脆弱的轮子。**
